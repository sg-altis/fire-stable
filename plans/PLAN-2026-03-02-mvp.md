# MVP Plan — fire-stable

## Goal

Turn the manual 15-tmux-session workflow into a dispatched, observable, cost-tracked pipeline. Human goes from operator to PM.

## Success metric

Within 2 weeks: human can `make dispatch` a feature to an agent and get a PR back, with cost tracked and Telegram escalation working.

## Phases

### Phase 1: Agent registry + butler integration (days 1–3)

**Deliverables:**
- `agents` table in butler's SQLite
- `cost_log` table in butler's SQLite
- Make targets: `make agent-add`, `make agent-status`, `make agent-cost`
- Agent CRUD in `butler/db.py`

**Why butler, not fire-stable?** Butler already has the SQLite infra, Make targets, TUI, and web dashboard. Adding agent tables there means the dashboard gets agent visibility for free. fire-stable is the design/docs repo; butler is where the code lives (for now).

**Acceptance criteria:**
- `make agent-add ID=opus-1 MODEL=claude-opus-4-6 VM=local` works
- `make agent-status` shows all agents with current state
- `make agent-cost` shows today's spend by agent
- TUI shows agent section (even if read-only initially)

### Phase 2: Dispatch command (days 3–5)

**Deliverables:**
- `make dispatch FEATURE=<idea>/<feature> AGENT=<id>` command
- Reads feature file from in3eption, validates it's accepted
- Creates butler TODO linked to feature
- Marks agent as working
- Starts tmux session with feature context injected

**Acceptance criteria:**
- Dispatching a feature to an agent starts a Claude Code session
- The session has the feature file, repo access, and branch name
- Agent registry shows the agent as working on that feature
- Butler TODO exists linked to the feature

### Phase 3: Telegram escalation (days 5–8)

**Deliverables:**
- Telegram bot created (BotFather)
- Group with topics enabled, thread-per-agent
- `butler/telegram.py` module
- Make targets: `make tg-alert`, `make tg-digest`
- Escalation message template documented

**Acceptance criteria:**
- Bot posts to agent thread when dispatched
- Agent can trigger escalation (manual for now: `make tg-alert AGENT=opus-1 MSG="..."`)
- Daily digest posts at EOD (agent count, PRs, cost)
- Cost threshold alerts fire when caps hit

### Phase 4: Cost tracking (days 8–10)

**Deliverables:**
- `cost_log` table populated (manual reporting initially)
- `make agent-cost-log AGENT=opus-1 TOKENS_IN=50000 TOKENS_OUT=10000`
- Cost aggregation queries (per-agent, per-day, per-feature)
- Cost cap enforcement (auto-pause + Telegram alert)
- Web dashboard cost view

**Acceptance criteria:**
- `make agent-cost` shows accurate daily spend
- Cost cap triggers auto-pause + Telegram alert
- Web dashboard shows cost chart (even simple)

### Phase 5: Dashboard polish (days 10–14)

**Deliverables:**
- Agent section in butler TUI (status, current work, cost)
- Agent section in butler web admin (status, cost, session access link)
- Feature pipeline view (proposed → accepted → in-progress → pr-open → merged)
- Cost history chart (7-day)

**Acceptance criteria:**
- TUI shows agents at a glance
- Web admin shows full agent detail
- Human can observe the entire pipeline from one screen

## Budget

```
Phase 1-5 development:
  ~40 Opus agent-hours × $15/hr    = ~$600
  ~20 Sonnet agent-hours × $3/hr   = ~$60
  Human review time: ~15 hours
  ────────────────────────────────────────
  Total: ~$700 + human time

Ongoing (once running):
  Per Phase 0 economics: $1,500–4,000/mo
  Savings: ~$600/mo in wasted cycles + 30 hr/mo human time
  Break-even: month 1
```

## Risks

1. **Scope creep** — This plan is already 5 phases. Ruthlessly cut to keep the 2-week timeline.
2. **Butler coupling** — Building into butler's SQLite is pragmatic but couples fire-stable's design to butler's architecture. Accept this for MVP; refactor later if needed.
3. **Token counting accuracy** — Manual reporting is lossy. Automated counting needs a wrapper around API calls. Accept manual for MVP.
4. **Telegram bot reliability** — Bot API is simple but needs error handling. Start with happy path only.

## Non-goals (for MVP)

- Cloud VMs (local tmux only)
- Auto-scaling
- Model selection automation (human picks model)
- Multi-repo dispatch (one repo per feature for now)
- in3eption feature file auto-sync (manual `make dispatch` for now)
